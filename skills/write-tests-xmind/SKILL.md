# Write Tests XMind Skill

从 XMind 思维导图生成测试用例和任务文档的自动化 skill。

## 用途

读取 `testcases/design/` 中的 XMind 文件，根据测试用例规范自动生成测试用例代码和任务文档。

## 适用场景

- 从 XMind 设计文件生成测试用例
- 自动化测试用例编写流程
- 创建测试任务文档
- 标准化测试用例格式

## 工作流程

### 1. 读取 XMind 文件
- 从 `testcases/design/` 目录读取指定的 XMind 文件
- 使用 xmind-mcp 工具解析 XMind 结构

### 2. 解析测试用例规范
- 读取 `testcases/README.md` 中的测试用例规范
- 识别测试方法（KDT 或 DDT）
- 提取用例格式要求

### 3. 生成测试用例
- 根据规范生成符合格式的测试用例
- 支持 KDT（关键字驱动）和 DDT（数据驱动）两种格式
- 保存到 `testcases/tests/` 目录

### 4. 生成任务文档
- 创建 `test-task.{需求编号}.md` 文档（文件名固定，不允许更改）
- 列出所有测试任务
- 保存到 `.kiro/specs/test-task.{需求编号}.md` 目录（路径和文件名固定）

## 使用 xmind-mcp 工具

### 读取 XMind 文件
使用 `xmind-mcp` 工具读取 XMind 文件内容。

**参数**：
- `file_path`: XMind 文件的绝对路径

**返回**：
- XMind 文件的结构化内容（包含所有 Sheet）
- 节点层级关系
- 主题和子主题信息

**多 Sheet 处理**：
- XMind 文件可能包含多个 Sheet（工作表）
- 每个 Sheet 通常代表一个独立的功能模块或测试场景

### 分析思维导图结构
使用 `xmind-mcp` 工具分析 XMind 结构。

**参数**：
- `file_path`: XMind 文件的绝对路径

**返回**：
- 节点统计信息
- 最大层级深度
- 结构分析结果

## 输出文件命名规范

### 测试用例文件
- 格式：`test_{module}_{scenario}.{需求编号}.test.md`
- 示例：`test_payment_overhead_rate.44470.test.md`
- 说明：相同需求的所有测试用例合并在一个文件中

### 任务文档
- **文件名（固定）**：`test-task.{需求编号}.md`（不允许使用其他文件名）
- **位置（固定）**：`.kiro/specs/test-task.md`
- 格式**严格按照**： `references\test-task-template.md`
- 示例参考： `references\test-task-example.md`
- **重要**：任务文档的文件名必须固定为 `test-task.{需求编号}.md`，不能根据 XMind 文件名变化

## 任务文档结构

```markdown
# 测试任务

## 概述
从 XMind 文件生成的测试任务列表

## 测试用例清单

### 需求：[需求编号] - [需求名称]

- [ ] 测试场景：[场景名]
  - 优先级: High
  - 方法: 场景法
  - 文件: test_module_scenario.需求编号.test.md
  - 用例数量: X 个

## 执行计划

1. 执行 Critical 优先级测试
2. 执行 High 优先级测试
3. 执行 Medium 优先级测试
4. 执行 Low 优先级测试
```

## 执行步骤

### 完整流程

1. **指定 XMind 文件**
   ```
   请使用 testcases/design/[文件名].xmind
   ```

2. **读取并解析 XMind**
   - 使用 xmind-mcp 读取文件
   - 解析思维导图结构
   - 识别测试用例节点

3. **读取测试规范**
   - 读取 testcases/README.md
   - 确认用例格式要求
   - 确认测试方法

4. **澄清需求编号（重要）**
   - 在生成测试用例之前，检查 XMind 内容中是否包含需求编号
   - 如果 XMind 中没有明确的需求编号，使用 userInput 工具询问用户
   - 询问格式：
     ```
     **需求编号确认**
     
     我在 XMind 中没有发现明确的需求编号。
     
     请提供需求编号（例如：PRODIGY-001、REQ-12345）：
     ```
   - 等待用户提供需求编号后再继续
   - 需求编号将用于：
     - 测试用例文件命名：`test_{module}_{scenario}.{需求编号}.test.md`
     - 测试用例编号：`TC-{需求编号}-001`
     - 任务文档标题和内容
   - 如果 XMind 中已有明确的需求编号，直接使用，无需询问

5. **澄清不清晰的内容（重要）**
   - 在生成测试用例之前，先分析 XMind 内容
   - 识别不清晰的术语、缩写或业务逻辑
   - 使用 userInput 工具询问用户以澄清疑问
   - 等待用户确认后再继续生成用例
   - 常见需要澄清的内容：
     - 业务术语的具体含义
     - 缩写的完整形式
     - 复杂的业务逻辑或规则
     - 状态转换的具体条件
     - 字段或功能的预期行为

6. **检查已有文件（重要）**
   - 在生成测试用例之前，检查 `testcases/tests/` 目录是否已存在相同需求编号的测试用例文件
   - 检查 `.kiro/specs/test-task.md` 是否已存在
   - 如果文件已存在：
     - **只能更新（追加或修改）现有文件**
     - **严禁删除已有的测试用例文件或 test-task.md 文件**
     - 使用 fsAppend 追加新的测试用例到现有文件
     - 使用 strReplace 更新现有内容
     - 保留所有已有的测试用例和任务
   - 如果文件不存在：
     - 可以创建新文件
     - 使用 fsWrite 创建新的测试用例文件和任务文档
   - **关键规则**：
     - 已有文件只能更新，不能删除
     - 已有测试用例只能修改或追加，不能删除
     - 已有任务只能更新或追加，不能删除

7. **生成测试用例**
   - 检查 `testcases/tests/` 目录是否已存在相同需求编号的测试用例文件
   - 如果文件已存在：
     - 使用 fsAppend 追加新的测试用例到现有文件
     - 保留所有已有的测试用例
   - 如果文件不存在：
     - 使用 fsWrite 创建新的测试用例文件
   - 根据规范生成测试用例代码
   - 应用正确的格式（KDT 或 DDT）
   - 基于用户澄清的内容生成准确的测试用例
   - 保存到 testcases/tests/

8. **自检覆盖率（重要）**
   - 读取生成的测试用例文件
   - 对比 XMind 中的所有测试场景和节点
   - 识别未覆盖的场景或逻辑
   - 如果发现遗漏，补充生成缺失的测试用例
   - 重复自检直到完全覆盖 XMind 内容
   - 自检要点：
     - 检查所有主要场景是否都有对应的测试用例
     - 检查所有状态组合是否都被覆盖
     - 检查所有配置开关的不同取值是否都有测试

9. **生成或更新任务文档**
   - 检查 `.kiro/specs/test-task.{需求编号}.md` 是否已存在
   - 如果文件已存在：
     - 使用 fsAppend 追加新的任务到现有文档
     - 保留所有已有的任务
   - 如果文件不存在：
     - 使用 fsWrite 创建新的任务文档
   - 创建 test-task.md（文件名固定，不能更改）
   - 列出所有测试任务
   - 保存到 .kiro/specs/test-task.{需求编号}.md

10. **格式自检（关键）**
   - 读取生成的测试用例文件和任务文档
   - 对比 `references/test_template.md` 检查测试用例格式
   - 对比 `references/test-task-template.md` 检查任务文档格式
   - 识别格式不符合的地方
   - 如果发现格式问题，立即修正
   - 重复自检直到格式完全符合模板要求
   - 格式检查要点：
     - 测试用例文件头部是否包含"测试套件信息"
     - 测试用例是否包含所有必需字段（Priority, Method, Precondition, Test Steps, Expected Result）
     - 任务文档是否包含所有必需章节（概述、测试用例集清单、统计信息、执行计划、测试方法分布、注意事项、相关文档）
     - 任务文档中的 checkbox 格式是否正确
     - 测试用例编号格式是否正确

11. **生成操作日志（重要）**
   - 如果更新了 test-task.md 或测试用例文件，必须生成日志文件
   - 日志文件命名格式：`write-test-xmind.{yy-mm-dd_hh-mm-ss}.log`
   - 日志文件保存位置：`.kiro/specs/logs/`
   - 日志内容必须包含：
     - 操作时间
     - XMind 文件名
     - 需求编号
     - 操作类型（新建/更新）
     - 本次操作的总结
     - 所有新增或更新的测试用例编号列表
     - 测试用例文件路径
     - 任务文档文件路径
   - 日志格式参考：
     ```
     ==========================================
     Write-Tests-XMind 操作日志
     ==========================================
     
     操作时间: 2026-03-11 14:30:25
     XMind 文件: testcases/design/Prodigy Dashboard.xmind
     需求编号: PRODIGY-001
     操作类型: 新建
     
     ==========================================
     操作总结
     ==========================================
     
     从 XMind 文件成功生成 30 个测试用例，覆盖以下功能模块：
     - 用户登录验证
     - Lifecycle/Role 显示
     - Breadcrumb 组件
     - Top Navigation
     - Applications 权限控制
     
     ==========================================
     测试用例列表
     ==========================================
     
     新增测试用例（30 个）：
     1. TC-PRODIGY-001-001: 用户登录成功
     2. TC-PRODIGY-001-002: 用户登录失败
     3. TC-PRODIGY-001-003: Lifecycle 显示正确
     ...
     30. TC-PRODIGY-001-030: 无权限应用不显示
     
     ==========================================
     文件信息
     ==========================================
     
     测试用例文件: testcases/tests/test_prodigy_dashboard.PRODIGY-001.test.md
     任务文档文件: .kiro/specs/test-task.md
     
     ==========================================
     ```

## 注意事项

- 测试步骤应该清晰明确
- 文件名使用小写和下划线
- 任务文档使用 Markdown checkbox 格式
- **在生成测试用例前，必须先澄清不清晰的内容**
- 使用 userInput 工具询问用户以确认术语、逻辑或业务规则
- 只有在用户确认后才开始生成测试用例
- 如果 XMind 内容清晰明确，可以直接生成测试用例
- **生成测试用例后，必须进行自检，确保完全覆盖 XMind 内容**
- 如果发现遗漏，立即补充缺失的测试用例
- 重复自检直到覆盖率达到 100%
- **生成所有文件后，必须进行格式自检，确保符合模板要求**
- 测试用例格式必须严格遵循 `references/test_template.md`
- 任务文档格式必须严格遵循 `references/test-task-template.md`
- 如果发现格式问题，立即修正
- 重复格式自检直到完全符合模板
- **完成所有操作后，必须生成操作日志**
- 日志文件保存在 `testcases/logs/` 目录
- 日志文件名格式：`write-test-xmind.{yy-mm-dd_hh-mm-ss}.{loginUser}.log`
- 日志必须包含操作总结和所有测试用例编号

## 澄清内容的示例

### 需要澄清的情况

1. **不清晰的术语或缩写**
   - 示例："IG"、"SDV"、"CRF" 等缩写
   - 询问："请确认 'IG' 是指什么？是 Item Group 吗？"

2. **复杂的业务逻辑**
   - 示例："表单在 site 1 做了 sign 之后，在 site 2 之后，如果用户没有 site 1 的权限就不能对这个表单做 sign 了"
   - 询问："请确认这个逻辑：用户需要有表单签名时所在 Site 的权限才能 Unsign，对吗？"

3. **状态转换条件**
   - 示例："ig=incomplete+softlock/sign"
   - 询问："请确认 'ig=incomplete+softlock/sign' 表示 IG 状态为 Incomplete 且同时有 Softlock 或 Sign 状态，对吗？"

4. **字段或功能的预期行为**
   - 示例："Data-Driven Form Updates=yes-->会取消softlock/sign的状态"
   - 询问："请确认当 'Data-Driven Form Updates' 配置为 Yes 时，任何数据变更都会取消 Softlock/Sign 状态，对吗？"

### 询问方式

使用 userInput 工具，格式如下：
```
**我在 XMind 中发现以下内容需要澄清：**

1. [术语/逻辑描述]
   - 我的理解：[你的理解]
   - 请确认是否正确

2. [另一个需要澄清的内容]
   - 我的理解：[你的理解]
   - 请确认是否正确

请确认以上理解是否正确，或提供正确的解释。
```

## 自检覆盖率流程

### 自检步骤

1. **读取生成的测试用例文件**
   - 使用 readFile 工具读取已生成的测试用例文件
   - 提取所有测试用例的标题和覆盖的场景

2. **对比 XMind 内容**
   - 重新分析 XMind 的结构和内容
   - 列出所有需要测试的场景、状态组合、配置项
   - 识别 XMind 中的关键测试点

3. **识别遗漏的场景**
   - 对比已生成的测试用例和 XMind 内容
   - 找出未覆盖的场景、状态组合或配置
   - 列出所有遗漏的测试点

4. **补充缺失的测试用例**
   - 为每个遗漏的场景生成测试用例
   - 使用 fsAppend 工具追加到测试用例文件
   - 确保新增用例符合格式规范

5. **重复自检**
   - 再次读取更新后的测试用例文件
   - 验证是否还有遗漏
   - 如果仍有遗漏，继续补充
   - 直到覆盖率达到 100%



### 自检要点

**场景覆盖检查**：
- 所有主要业务场景是否都有测试用例
- 所有异常场景是否都有测试用例
- 所有边界条件是否都有测试用例

### 自检输出格式

在自检过程中，应该输出以下信息：

```
## 自检报告

### 已覆盖的场景（X 个）
1. [场景描述] - 测试用例 TC-XXXXX-XXX
2. [场景描述] - 测试用例 TC-XXXXX-XXX
...

### 未覆盖的场景（Y 个）
1. [场景描述] - 需要补充
2. [场景描述] - 需要补充
...

### 覆盖率：X / (X + Y) = Z%

### 下一步行动
- 补充 Y 个缺失的测试用例
- 重新进行自检验证
```

### 自检完成标准

- 覆盖率达到 100%
- XMind 中的所有场景都有对应的测试用例
- 所有状态组合都被测试
- 所有配置项的不同取值都被测试
- 所有操作类型都被测试
- 所有表单类型都被测试

## 格式自检流程

### 格式检查步骤

1. **读取模板文件**
   - 使用 readFile 工具读取 `references/test_template.md`
   - 使用 readFile 工具读取 `references/test-task-template.md`
   - 理解模板的结构和必需字段

2. **检查测试用例格式**
   - 读取生成的测试用例文件
   - 对比模板检查以下内容：
     - 文件头部是否包含"测试套件信息"章节
     - 是否包含：功能模块、需求编号、测试目标
     - 每个测试用例是否包含：用例编号、Priority、Method、Precondition、Test Steps、Expected Result
     - Test Steps 是否使用编号列表格式
     - Expected Result 是否使用列表格式
     - 用例之间是否使用 `---` 分隔符

3. **检查任务文档格式**
   - 读取生成的任务文档文件
   - 对比模板检查以下内容：
     - 是否包含"概述"章节（XMind 文件名、生成时间）
     - 是否包含"测试用例集清单"章节
     - 清单中的 checkbox 格式是否正确（`- [ ]` 或 `- [x]`）
     - 是否包含"统计信息"章节（总数、各优先级数量）
     - 是否包含"执行计划"章节（按优先级分阶段）
     - 是否包含"测试方法分布"章节
     - 是否包含"注意事项"章节
     - 是否包含"相关文档"章节

4. **识别格式问题**
   - 列出所有不符合模板的地方
   - 标记缺失的必需字段
   - 标记格式不正确的内容

5. **修正格式问题**
   - 使用 strReplace 工具修正测试用例格式
   - 使用 strReplace 工具修正任务文档格式
   - 确保修正后的内容完全符合模板

6. **重复格式自检**
   - 再次读取修正后的文件
   - 验证是否还有格式问题
   - 如果仍有问题，继续修正
   - 直到格式完全符合模板

### 格式自检要点

**测试用例格式检查**：
- 文件头部必须包含"测试套件信息"
- 每个测试用例必须包含所有必需字段
- Test Steps 必须使用编号列表
- Expected Result 必须使用列表格式
- 用例之间必须使用 `---` 分隔

**任务文档格式检查**：
- 必须包含所有必需章节
- Checkbox 格式必须正确
- 统计信息必须准确
- 执行计划必须按优先级分阶段
- 测试方法分布必须统计所有方法

### 格式自检输出格式

在格式自检过程中，应该输出以下信息：

```
## 格式自检报告

### 测试用例格式检查
- [✓] 文件头部包含测试套件信息
- [✓] 所有用例包含必需字段
- [✗] Test Steps 格式不正确 - 需要修正
- [✓] Expected Result 格式正确
- [✓] 用例分隔符正确

### 任务文档格式检查
- [✓] 包含概述章节
- [✓] 包含测试用例集清单
- [✗] Checkbox 格式不正确 - 需要修正
- [✓] 包含统计信息
- [✓] 包含执行计划
- [✓] 包含测试方法分布
- [✓] 包含注意事项
- [✓] 包含相关文档

### 需要修正的问题（2 个）
1. 测试用例 Test Steps 格式不正确
2. 任务文档 Checkbox 格式不正确

### 下一步行动
- 修正 2 个格式问题
- 重新进行格式自检验证
```

### 格式自检完成标准

## 操作日志生成流程

### 日志生成时机

- 在完成所有测试用例生成和格式自检后
- 只要更新了 test-task.md 或测试用例文件，就必须生成日志
- 日志生成是最后一步，不可省略

### 日志文件规范

**文件命名**：
- 格式：`write-test-xmind.{yy-mm-dd_hh-mm-ss}.{loginUser}.log`
- 示例：`write-test-xmind.26-03-11_14-30-25.DiWang.log`
- 使用当前时间生成文件名

**保存位置**：
- 目录：`testcases/logs/`
- 如果目录不存在，先创建目录
- 完整路径示例：`testcases/logs/write-test-xmind.26-03-11_14-30-25.DiWang.log`

### 日志内容结构

日志文件必须包含以下章节：

1. **标题和分隔线**
   ```
   ==========================================
   Write-Tests-XMind 操作日志
   ==========================================
   ```

2. **基本信息**
   - 操作时间：格式 `YYYY-MM-DD HH:MM:SS`
   - 操作用户： 当前登陆Kiro的账号名
   - XMind 文件：相对路径
   - 需求编号：如 PRODIGY-001
   - 操作类型：新建 或 更新

3. **操作总结**
   - 生成或更新的测试用例总数
   - 覆盖的功能模块列表
   - 主要测试场景概述

4. **测试用例列表**
   - 新增测试用例：列出所有新增的用例编号和标题
   - 更新测试用例：列出所有更新的用例编号和标题（如果是更新操作）
   - 格式：`序号. TC-{需求编号}-{编号}: {用例标题}`

5. **文件信息**
   - 测试用例文件路径
   - 任务文档文件路径

### 日志生成步骤

1. **收集信息**
   - 获取当前时间
   - 获取当前登陆 Kiro 用户名
   - 记录 XMind 文件名
   - 记录需求编号
   - 确定操作类型（新建或更新）
   - 统计测试用例数量
   - 提取所有测试用例编号和标题

2. **生成日志内容**
   - 按照模板格式组织内容
   - 填充所有必需信息
   - 确保格式整齐美观

3. **创建日志目录**
   - 检查 `testcases/logs/` 目录是否存在
   - 如果不存在，使用 fsWrite 创建目录结构

4. **写入日志文件**
   - 使用 fsWrite 创建日志文件
   - 文件名使用当前时间戳
   - 保存到 `testcases/logs/` 目录

5. **确认日志生成**
   - 读取日志文件验证内容
   - 向用户报告日志文件路径
